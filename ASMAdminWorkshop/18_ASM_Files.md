## Lesson 18 - ASM Files (in which filenames become both a science and a low-key identity crisis)

And look, this chapter is about how ASM names files, how humans reference files, and how to see what actually exists without guessing your way through plus-sign hierarchies.

By the end of this lesson, you should be able to:

- Explain what a fully qualified ASM file name is and why only ASM generates it
- Distinguish fully qualified names, user file references, templates, and incomplete names
- Use OMF-style parameters for multi-file creation patterns
- Query ASM file and alias metadata from dynamic performance views
- Use ASMCMD to inspect file-level details that map to the same underlying views

---

## 1. Fully Qualified ASM File Names

Fully qualified ASM file names are ASM-generated internal identities.

Important rules:

- You do not manually create these names
- ASM guarantees uniqueness
- They are used internally for file identification/management

You will mostly encounter them in:

- Dynamic performance views
- Database metadata output
- Diagnostic output

So yes, you can read them. No, you should not try to hand-craft them like artisanal filenames.

---

## 2. Typical Fully Qualified Name Shape

A common structure looks like:

```text
+<diskgroup>/<db_unique_name>/<file_type>/<tag>.<file_number>.<incarnation>
```

Common path elements:

- Disk group (`+DATA`, `+FRA`, etc.)
- Database identity directory
- File class (`DATAFILE`, `CONTROLFILE`, `ONLINELOG`, `PARAMETERFILE`, ...)
- Tag/name component
- Numeric file number
- Incarnation/version component

Example classes from this lesson context:

- Data file
- SPFILE
- Control file
- Redo log file
- FRA-side backup/copy objects

---

## 3. Human-Facing Naming Styles You Actually Use

You typically create files with one of these patterns:

1. Explicit alias-like path:
   - Full disk group + directory + filename you supply
2. Alias + template:
   - Your path plus template policy
3. Incomplete name:
   - Only disk group (rest generated by ASM/OMF behavior)
4. Incomplete + template:
   - Minimal specification with template-driven policy

The fully qualified internal name is still generated by ASM either way.

---

## 4. Single-File vs Multi-File Creation Behavior

Practical behavior:

- Explicit alias styles are common for targeted single-file operations
- Incomplete naming is useful for system-generated file creation
- OMF/multi-destination configs are ideal when creating sets (for example redo members)

Example OMF-style redo setup:

```sql
ALTER SYSTEM SET DB_CREATE_ONLINE_LOG_DEST_1 = '+DATA';
ALTER SYSTEM SET DB_CREATE_ONLINE_LOG_DEST_2 = '+FRA';

ALTER DATABASE ADD LOGFILE;
```

Result:

- Oracle creates member files in each configured destination
- Internal ASM names are generated automatically

This is how you let the database do the boring naming work while you keep your blood pressure.

---

## 5. ASM Aliases vs "Alias-Like" File References

Transcript nuance worth clarifying:

- The path you specify during creation is a user reference style
- ASM also supports explicit alias objects (like symbolic-link style indirection)

They are related in workflow but not identical concepts.

You will manage true aliases more directly in directory/alias-focused chapters.

---

## 6. Viewing Alias Metadata

Core view:

- `V$ASM_ALIAS`

What it provides:

- One row per alias/directory-style element in ASM namespace hierarchy
- Relationship data for traversing file/directory structures

Scope note:

- Querying from ASM instance gives broad mounted-group context
- Querying from database instance is scoped to what that database can see/use

ASMCMD equivalent:

- `ls` and related listing flows expose comparable information paths

---

## 7. Viewing File Metadata

Core view:

- `V$ASM_FILE`

What it gives:

- One row per ASM file in mounted disk groups (ASM context)
- File type, redundancy, size/block characteristics, and related identifiers

Useful joins:

- `V$ASM_FILE` + `V$ASM_ALIAS` + `V$ASM_DISKGROUP`

This gives you both identity and readable namespace context in one query.

---

## 8. ASMCMD File Inspection

ASMCMD listing commands expose details similar to those views, including:

- File type
- Redundancy
- Block size and count
- Byte usage / space metrics

Example path listing:

```bash
asmcmd ls -l +DATA/ORCLCDB/DATAFILE
```

If it feels like Linux `ls` with Oracle storage metadata bolted on, that is exactly what it is.

---

## 9. Practical Naming Strategy

Recommended approach:

1. Use incomplete/OMF naming where automation and consistency matter
2. Use explicit naming selectively when operational clarity requires it
3. Rely on views/ASMCMD for truth, not memory

Because once filenames include file numbers and incarnations, "I know where that file is" becomes a dangerous sentence.

---

## 10. Key Takeaways

- Fully qualified ASM names are system-generated internal identities.
- User workflows use explicit, template-based, or incomplete naming inputs.
- OMF-style configuration is ideal for clean multi-file generation patterns.
- `V$ASM_ALIAS` and `V$ASM_FILE` are the core inspection views.
- ASMCMD provides a Linux-like interactive layer on top of the same metadata reality.
